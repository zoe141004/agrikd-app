# Contributing to AgriKD

Plant leaf disease recognition using Knowledge Distillation (ViT-Base to
MobileNetV2). Includes a Flutter mobile app, an MLOps pipeline, a React admin
dashboard, and Jetson edge deployment. This guide reflects the Phase 2 layout.

---

## 1. Prerequisites

| Tool | Minimum Version | Purpose |
|------|-----------------|---------|
| Git | 2.30+ | Version control |
| Flutter SDK | 3.41.4+ | Mobile application |
| Python | 3.10+ | MLOps pipeline |
| Node.js | 18+ | Admin dashboard |
| Android Studio or VS Code | Latest stable | IDE with Flutter/Dart plugin |
| JetPack SDK with TensorRT | *(Jetson only)* | Edge deployment |

Always activate `venv_mlops/` before running pipeline scripts; the system
Python may carry outdated PyTorch versions.

## 2. Quick Setup

### Automated (Windows)

```bash
git clone https://github.com/<org>/agrikd.git && cd agrikd
setup_windows_dev.bat
```

### Manual

```bash
git clone https://github.com/<org>/agrikd.git && cd agrikd
cd mobile_app && flutter pub get && cd ..                   # Flutter
python -m venv venv_mlops                                   # MLOps venv
source venv_mlops/bin/activate                              # Win: venv_mlops\Scripts\activate
pip install -r mlops_pipeline/requirements.txt
cd admin-dashboard && npm install && cd ..                   # Admin dashboard
```

## 3. Environment Variables

```bash
cp .env.example .env       # fill in your credentials
python sync_env.py
```

`sync_env.py` reads the root `.env` (single source of truth) and generates:
- `mobile_app/.env` -- copy for Flutter (assets cannot reference parent dirs)
- `admin-dashboard/.env` -- VITE_ prefixed variables for Vite
- `jetson/config/config.json` -- Supabase credentials in the sync block

### Variable Reference

| Variable | Description | Used By |
|----------|-------------|---------|
| `SUPABASE_URL` | Supabase project URL | Flutter, Admin, Jetson |
| `SUPABASE_ANON_KEY` | Supabase anonymous key | Flutter, Admin, Jetson |
| `GOOGLE_WEB_CLIENT_ID` | Google OAuth client ID | Flutter |
| `SUPABASE_SERVICE_ROLE_KEY` | Service-level access | CI/CD workflows |
| `GDRIVE_CREDENTIALS_DATA` | GDrive service account JSON | DVC workflows |
| `SENTRY_DSN` | Sentry error tracking DSN | Flutter, Admin Dashboard |

### CI/CD Secrets (GitHub)

| Secret | Workflows |
|--------|-----------|
| `SUPABASE_URL` | ci.yml, release.yml, model-pipeline.yml |
| `SUPABASE_ANON_KEY` | ci.yml, release.yml |
| `SUPABASE_SERVICE_ROLE_KEY` | model-pipeline.yml, dataset-upload.yml, export-data.yml |
| `GDRIVE_CREDENTIALS_DATA` | All DVC workflows |
| `SENTRY_DSN` | ci.yml, release.yml |
| `VERCEL_DEPLOY_HOOK` | deploy.yml |

In CI/CD production builds, values are injected via `--dart-define` so the
release APK never ships a `.env` file. Locally, `flutter_dotenv` reads
`mobile_app/.env` as a fallback.

## 4. Running the App

### Flutter (Mobile) -- all commands from `mobile_app/`

```bash
cd mobile_app
flutter run                # Debug -- auto-loads .env via flutter_dotenv
flutter run --release      # Release -- requires --dart-define values
```

### Admin Dashboard

```bash
cd admin-dashboard
npm run dev                # predev hook auto-syncs .env from root
```

Vite dev server includes security headers configured in `vite.config.js`.

### MLOps Pipeline

```bash
source venv_mlops/bin/activate
cd mlops_pipeline/scripts
python run_pipeline.py --config ../configs/tomato.json
python run_pipeline.py --config ../configs/burmese_grape_leaf.json
```

## 6. Development Workflow

### Branch Naming

| Prefix | Purpose |
|--------|---------|
| `feature/` | New functionality |
| `fix/` | Bug fixes |
| `docs/` | Documentation only |
| `refactor/` | Code restructuring |
| `ci/` | CI/CD changes |

### Commits

Conventional-commit style:

```
feat: add confidence threshold slider to scan screen
fix: correct class mapping for Burmese Grape Leaf
docs: update benchmark results table
ci: cache DVC pulls in model-pipeline workflow
```

### Pull Requests

1. Branch from `main`, make changes, ensure CI passes.
2. Open a PR targeting `main`; request review from at least one maintainer.

`validate_models.py` may report FAIL on random noise inputs due to fresh model
instances with different batch-norm statistics. This is not a real failure.

```bash

```

## 8. Security

### Release Build Obfuscation

Production builds must include obfuscation and split debug info:

```bash
cd mobile_app
flutter build apk --obfuscate --split-debug-info=build/debug-info \
  --dart-define=SUPABASE_URL=... \
  --dart-define=SUPABASE_ANON_KEY=... \
  --dart-define=SENTRY_DSN=...
```

Upload `build/debug-info` symbol maps to Sentry for readable crash reports.

### Sentry Error Tracking

- **Flutter**: `sentry_flutter` in `mobile_app/lib/main.dart`; reads `SENTRY_DSN`.
- **Admin Dashboard**: `@sentry/react` in `admin-dashboard/src/main.jsx`; reads `VITE_SENTRY_DSN`.

Both require a valid `SENTRY_DSN` in root `.env` (local) or GitHub Secrets (CI).

### Row-Level Security

Run `database/verify_rls_policies.sql` against Supabase after any migration or
policy change to confirm RLS is correctly applied.

### Sensitive Files

| Path | Status | If Exposed |
|------|--------|------------|
| `.dvc/gdrive-sa.json` | Gitignored | Rotate GDrive SA key immediately |
| `jetson/config/config.json` | Gitignored | Recreate from `config.example.json` |
| `.env` | Gitignored | Rotate all keys and Sentry DSN |

### Jetson Flask API

Rate limited at 30 req/min per client; 10 MB max upload. Do not relax without review.

### Admin Dashboard Headers

`vite.config.js` sets security headers (CSP, X-Frame-Options, etc.). Ensure
your production host applies equivalent headers.
