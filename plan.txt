## Phase 1: Jetson Blockers (6 fixes, 7 files)

> **Goal**: Make Jetson edge deployment crash-free and functionally correct.

### Fix 1.1 — C1: Inference Worker Queue (eliminates CUDA threading bug)

**Problem**: `pycuda.autoinit` (`inference.py:36`) binds CUDA context to the importing thread. `predict()` uses shared pre-allocated buffers (`inference.py:72-80`). Main thread (`main.py:174`) and Waitress threads (`health_server.py:163`) call `predict()` concurrently → buffer overwrite, CUDA corruption.

**Solution**: Create `InferenceWorkerPool` — a single worker thread owns the CUDA context and all buffers. All callers submit jobs via `queue.Queue`, receive results via `concurrent.futures.Future`.

**Why Queue > push/pop+Lock**: push/pop still shares buffers (needs Lock anyway). Queue naturally serializes, zero context leak risk, cleaner architecture. Throughput: ~450 FPS (1-2ms inference + 0.1ms queue overhead) — more than sufficient.

**Files**:
- `jetson/app/inference.py` — Add `InferenceWorkerPool` class (~60 lines). Wraps existing `TensorRTInference`. Worker thread: `while True: job = queue.get() → result = engine.predict(frame) → future.set_result(result)`. Public API: `pool.submit(leaf_type, frame) → Future`.
- `jetson/app/main.py` — Replace `engines[leaf_type].predict(frame)` (line 174) with `pool.submit(leaf_type, frame).result()`. Pass `pool` instead of `engines` dict to health_server.
- `jetson/app/health_server.py` — Replace `engine.predict(frame)` (line 163) with `pool.submit(leaf_type, frame).result()`. Remove `_inference_lock` (no longer needed — Queue serializes). Accept `pool` parameter instead of `engines` dict.
- `jetson/app/gui_app.py` — **NO CHANGE** (runs as separate process with own CUDA context, confirmed at `gui_app.py:556`).

**Local-First impact**: NONE. Queue is in-process, no network dependency. `submit().result()` blocks the calling thread only for inference time (~2ms).

### Fix 1.2 — C2: SQLite threading lock

**Problem**: Single `sqlite3.Connection` (`database.py:14`, `check_same_thread=False`) shared across 6+ threads with no Python-level synchronization. WAL + busy_timeout provide some protection but Python's sqlite3 module is not guaranteed thread-safe on the same connection.

**Solution**: Add a single `threading.Lock()` wrapping all `execute()`+`commit()` pairs.

**Why single Lock (not connection pool)**: Deadlock impossible with one lock (no circular dependency). All DB operations are <1ms. Starvation risk negligible (sync's `mark_synced` is single UPDATE per call). Connection pool adds complexity for zero practical benefit at this scale.

**Files**:
- `jetson/app/database.py` — Add `self._lock = threading.Lock()`. Wrap every method body (`save_prediction`, `get_unsynced`, `mark_synced`, `get_stats`, `cleanup_old_records`, `close`) in `with self._lock:`. ~15 lines changed.

**Local-First impact**: NONE. Lock is in-process, no network. Maximum blocking: <1ms per operation.

### Fix 1.6 — NEW-1: Graceful shutdown (try/finally + daemon drain)

**Problem**: `main.py:149-190` has no try/finally around the main loop. Uncaught exception skips `camera.release()` and `db.close()`. Also, `db.close()` can execute while sync_engine daemon thread is mid-operation → `ProgrammingError` on closed database.

**Solution**: Wrap main loop in try/finally. Set a `shutdown_event` that sync_engine checks, with a brief drain timeout.

**Files**:
- `jetson/app/main.py` — Wrap lines 149-190 in `try: ... finally: camera.release(); db.close()`. Add `shutdown_event = threading.Event()` passed to SyncEngine. Signal handler sets `shutdown_event`. Before `db.close()`, call `sync_thread.join(timeout=5)` to drain.
- `jetson/app/sync_engine.py` — Accept `shutdown_event` in `__init__`. In `run()` loop, check `if self._shutdown_event.is_set(): break` instead of `while True`. This allows graceful exit.

### Fix 1.2 — C2: SQLite threading lock

**Problem**: Single `sqlite3.Connection` (`database.py:14`, `check_same_thread=False`) shared across 6+ threads with no Python-level synchronization. WAL + busy_timeout provide some protection but Python's sqlite3 module is not guaranteed thread-safe on the same connection.

**Solution**: Add a single `threading.Lock()` wrapping all `execute()`+`commit()` pairs.

**Why single Lock (not connection pool)**: Deadlock impossible with one lock (no circular dependency). All DB operations are <1ms. Starvation risk negligible (sync's `mark_synced` is single UPDATE per call). Connection pool adds complexity for zero practical benefit at this scale.

**Files**:
- `jetson/app/database.py` — Add `self._lock = threading.Lock()`. Wrap every method body (`save_prediction`, `get_unsynced`, `mark_synced`, `get_stats`, `cleanup_old_records`, `close`) in `with self._lock:`. ~15 lines changed.

**Local-First impact**: NONE. Lock is in-process, no network. Maximum blocking: <1ms per operation.

### Fix 1.5 — H2: Health server auth hardening (REVISED per user feedback)

### Fix 1.7 — Wake-Capture-Sleep Camera (user-requested optimization)

**Problem**: In periodic mode (interval 30min+), camera stays open continuously → Jetson overheats, sensor degradation, wasted power. Plant diseases develop slowly — no need for continuous video stream.

**Solution**: Change `camera.py` to open/capture/release per cycle. Current code ALREADY has lazy `_open()` + `release()` with `self.cap = None` check (`camera.py:15-36`). Add a `capture_single()` method that opens camera → skips 3 warm-up frames (auto-exposure settle) → captures 1 frame → releases immediately.

**camera.py currently:**
```
_open() → lazy open (check cap is None)
capture() → _open() then read one frame (cap stays open)
release() → explicit cleanup
```

**camera.py after fix:**
```
_open()          → same
capture()        → same (for interactive/GUI use, camera stays open)
capture_single() → NEW: _open() → skip 3 frames → read → release() → return frame
release()        → same
```

**Integration**:
- `main.py` periodic mode: Replace `camera.capture()` with `camera.capture_single()` (wake-capture-sleep per cycle)
- `main.py` manual mode: No camera use (API-only), no change needed
- `gui_app.py`: Uses CameraThread with continuous `capture()` — NO CHANGE (GUI needs live preview)
- `health_server.py`: No camera use (receives uploaded images), no change needed

**Buffer leak risk**: NONE. `cv2.VideoCapture.release()` calls V4L2 `VIDIOC_STREAMOFF` + `close(fd)` — kernel buffer is fully freed. Re-opening is ~100-200ms, negligible vs 30-minute interval.

**Files**:
- `jetson/app/camera.py` — Add `capture_single(self, warmup_frames=3)` method. After `_open()`, loop `warmup_frames` times calling `self.cap.read()` (discard), then capture final frame, then `self.release()`.
- `jetson/app/main.py` — Line 171: Replace `camera.capture()` with `camera.capture_single()` in periodic mode.
